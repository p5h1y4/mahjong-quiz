# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## User context

このリポジトリのユーザーは非エンジニアです。回答時は以下を守ること：

- 専門用語を使う場合は必ず簡単な補足説明を添える
- 手順を説明するときはステップごとに番号付きで、Xcodeの画面上のどこを操作するかを具体的に書く
- コード変更の理由を「何のためにやるのか」から先に説明する
- 選択肢がある場合はおすすめを明示し、理由を一言で添える
- エラーが出た場合は原因と対処法をセットで、操作手順として伝える



## Language

The app UI and AI prompts are in Japanese. Speech recognition is hardcoded to `ja-JP`.

## Workflow rules

- 3ステップ以上のタスクや設計判断が必要な場合はplan modeに入ること。途中で想定外の問題が出たら作業を止めて再計画する。
- 複雑な調査や並列可能な作業はsubagentに委任し、メインのコンテキストを軽く保つ。
- タスク完了前に必ずビルド成功を確認する。「動くはず」で終わらせない。
- ユーザーから修正指示を受けたら、同じミスを繰り返さないようメモリに記録する。
- 変更は必要最小限にとどめる。根本原因を特定し、一時的な回避策ではなく正しい修正を行う。
- バグ報告を受けたら、ユーザーに質問せず自力でログ・エラーを確認して修正する。
- **エラー修正の上限**: ビルドエラーの修正を3回試しても成功しない場合は作業を一時停止し、ユーザーに状況を報告して判断を仰ぐ。無限ループでコンテキストを浪費しない。
- **大きな変更前のセーフティネット**: plan modeに入る前や複数ファイルにまたがる変更の前に、ユーザーにgit commitを促す。ローカルcommitのみ行い、pushはユーザーの明示的な指示なしに実行しない。
- **変更履歴の記録（必須）**: ファイルを変更するたびに、必ずCHANGES.mdの先頭に変更内容を追記すること。バックアップファイル名・変更前後の内容・理由を記載する。小さな変更でも省略しない。
- **commitの粒度**: 1つの機能・修正が完成したらすぐにcommitする。複数の機能をまとめてcommitしない。
- **commitメッセージの形式**: 接頭辞をつける。`feat:` 新機能、`fix:` バグ修正、`docs:` ドキュメント変更、`style:` 見た目の調整、`refactor:` 動作を変えないコード整理。
- **ブランチ管理（必須）**: mainへの直接pushは禁止。必ずフィーチャーブランチを作成してPR経由でマージすること。
  - ブランチ名の形式: `feat/機能名`、`fix/バグ名`、`docs/内容` など
  - 手順: `git checkout -b feat/xxx` → commit → `git push -u origin feat/xxx` → `gh pr create` → `gh pr merge`
  - PRタイトルはcommitメッセージと同じ形式（`feat:` `fix:` など）
- **PR前の動作確認（必須）**: `gh pr create` の前に、ブラウザでローカルファイルを開いてクイズ開始・回答・終了・ランキング表示が正常に動くことを確認する。
